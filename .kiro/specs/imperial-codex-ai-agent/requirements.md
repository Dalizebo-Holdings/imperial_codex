# Requirements Document

## Imperial Codex AI Agent — Supabase Migration, AI Chat Assistant, Autonomous Background Agent, AI-Powered Strike Output Engine, and Vercel Deployment

---

## Introduction

This document specifies the requirements for extending the Imperial Codex v16 strategic operating system with five interconnected capabilities:

1. **Supabase Migration** — replace all filesystem-based persistence with a PostgreSQL-backed Supabase database, providing durable, queryable storage for audit logs, loop execution records, instruments, capital allocations, vault data, and agent conversation history.
2. **AI Chat Assistant** — a streaming GPT-4o-powered chat widget embedded in the dashboard, with tool-use access to all core Codex data, conversation persistence, and streamed responses via the Vercel AI SDK.
3. **Autonomous Background Agent** — a GPT-4o-powered Vercel Cron job that evaluates all 104 Recursive Loop conditions every 15 minutes, auto-triggers qualifying loops, generates a daily Strike Output summary, and posts webhook alerts for critical events.
4. **AI-Powered Strike Output Engine** — replace the template-based Strike Output Engine with Claude-powered generation that receives structured context and produces all five sections with real reasoning, validated against structural requirements, with template fallback on API failure.
5. **Vercel Deployment Configuration** — `vercel.json` with cron configuration, environment variable documentation, Supabase migration SQL files, and removal of the Netlify configuration.

The existing in-memory store (Pillars, OS Modules, Library — loaded from Markdown at startup) is preserved for read operations. All new persistence operations target Supabase exclusively.

## Glossary

- **Supabase** — the PostgreSQL-as-a-service platform used as the new persistence layer, accessed via `@supabase/supabase-js`.
- **Supabase_Client** — the singleton `@supabase/supabase-js` client instance used by all server-side database operations.
- **Supabase_Service_Client** — the elevated Supabase client initialised with `SUPABASE_SERVICE_ROLE_KEY`, used for operations that bypass Row Level Security (e.g., vault reads, cron agent writes).
- **RLS** — Row Level Security; Supabase's PostgreSQL-level access control policy applied to the `vault` table.
- **Migration_File** — a versioned SQL file under `/supabase/migrations/` that defines or alters the database schema.
- **Chat_Widget** — the streaming AI chat interface embedded in the Codex dashboard, accessible to all authenticated users.
- **Chat_Session** — a single continuous conversation between a user and the AI Chat Assistant, persisted as a row in `agent_conversations`.
- **Chat_Message** — a single turn (user or assistant) within a Chat_Session, persisted as a row in `agent_messages`.
- **AI_Tool** — a function exposed to the GPT-4o model via the Vercel AI SDK tool-use interface, allowing the model to query Codex data during a chat turn.
- **Background_Agent** — the autonomous GPT-4o process invoked by the Vercel Cron scheduler to evaluate loop conditions and generate daily summaries.
- **Cron_Job** — a scheduled invocation defined in `vercel.json` that triggers a Next.js API route at a fixed UTC schedule.
- **Loop_Evaluation** — the Background_Agent's assessment of whether a Recursive_Loop's trigger condition is satisfied by current system state.
- **Daily_Summary** — the Strike Output document generated by the Background_Agent at 06:00 UTC each day, summarising system state across all active OS Modules.
- **Webhook_Alert** — an HTTP POST payload sent to a configurable endpoint (Slack or email relay) when a critical Recursive_Loop is triggered by the Background_Agent.
- **Critical_Loop** — a Recursive_Loop whose definition includes a `critical: true` flag, indicating that its triggering warrants an immediate external alert.
- **Claude_Engine** — the Anthropic Claude model invoked via `@anthropic-ai/sdk` to generate Strike Output sections with real reasoning.
- **Structured_Context** — the JSON payload assembled by the Strike Output Engine and passed to Claude, containing relevant Pillars, OS Modules, Library entries, and capital allocation state.
- **Template_Fallback** — the original template-based Strike Output generation path, invoked when the Claude API is unavailable or returns an invalid response.
- **Strike_Validation** — the post-generation check that verifies a Claude-generated Strike Output contains exactly five sections with correct labels before the output is accepted.
- **Vercel_Cron** — Vercel's built-in cron scheduler, configured via `vercel.json`, that invokes Next.js API routes on a UTC schedule.
- **SUPABASE_URL** — environment variable holding the Supabase project URL.
- **SUPABASE_ANON_KEY** — environment variable holding the Supabase anonymous (public) API key.
- **SUPABASE_SERVICE_ROLE_KEY** — environment variable holding the Supabase service role key, granting elevated database access.
- **OPENAI_API_KEY** — environment variable holding the OpenAI API key used by both the Chat Assistant and Background Agent.
- **ANTHROPIC_API_KEY** — environment variable holding the Anthropic API key used by the Claude Engine.
- **VAULT_ENCRYPTION_KEY** — environment variable holding the 64-character hex key for AES-256-GCM vault encryption (carried over from v16).
- **SESSION_SECRET** — environment variable holding the `iron-session` cookie encryption secret (carried over from v16).
- **Vercel_AI_SDK** — the `ai` npm package providing streaming chat primitives, tool-use orchestration, and the `streamText` / `generateText` APIs used by both the Chat Assistant and Background Agent.

## Requirements

### Requirement 1: Supabase Database Schema and Migration Files

**User Story:** As an Admin, I want all persistence operations to target a Supabase PostgreSQL database defined by versioned migration files, so that the schema is reproducible, auditable, and deployable to any Supabase project.

#### Acceptance Criteria

1. THE Codex SHALL provide a `/supabase/migrations/` directory containing at least one Migration_File that creates all required tables before any application code attempts to write to the database.
2. THE Migration_File SHALL create an `audit_log` table with columns: `id` (UUID primary key, default `gen_random_uuid()`), `user_id` (text), `resource` (text), `clearance_level` (integer), `decision` (text), `timestamp` (timestamptz, default `now()`), and `details` (jsonb).
3. THE Migration_File SHALL create a `loop_execution_log` table with columns: `id` (UUID primary key), `loop_id` (text not null), `triggered_at` (timestamptz not null), `condition_matched` (text), `target_slug` (text), `output_action` (text), `outcome` (text), and `triggered_by` (text).
4. THE Migration_File SHALL create an `instruments` table with columns: `id` (text primary key, format `DH-RES-YYYY-NNN`), `title` (text), `issuing_authority` (text), `content` (text), `generated_at` (timestamptz not null), `status` (text default `'active'`), and `generated_by` (text).
5. THE Migration_File SHALL create an `instrument_registry` table with columns: `id` (text primary key), `title` (text), `issuing_authority` (text), `generated_at` (timestamptz not null), `status` (text), and `year` (integer generated always as `EXTRACT(YEAR FROM generated_at)` stored).
6. THE Migration_File SHALL create a `capital_allocations` table with columns: `id` (UUID primary key), `growth_pct` (numeric not null), `operational_pct` (numeric not null), `reserve_pct` (numeric not null), `approved_by` (text not null), `os_module_slug` (text not null), and `approved_at` (timestamptz not null).
7. THE Migration_File SHALL create a `capital_allocation_failures` table with columns: `id` (UUID primary key), `submitted_by` (text), `submitted_at` (timestamptz not null), `payload` (jsonb not null), and `validation_errors` (jsonb not null).
8. THE Migration_File SHALL create a `vault` table with columns: `id` (integer primary key default 1), `encrypted_envelope` (text not null), and `updated_at` (timestamptz not null), with a check constraint ensuring `id = 1` (single-row table).
9. THE Migration_File SHALL create an `agent_conversations` table with columns: `id` (UUID primary key), `user_id` (text not null), `title` (text), `created_at` (timestamptz default `now()`), and `updated_at` (timestamptz default `now()`).
10. THE Migration_File SHALL create an `agent_messages` table with columns: `id` (UUID primary key), `conversation_id` (UUID not null references `agent_conversations(id)` on delete cascade), `role` (text not null, check `role IN ('user', 'assistant', 'tool')`), `content` (text not null), `tool_calls` (jsonb), and `created_at` (timestamptz default `now()`).
11. THE Migration_File SHALL enable Row Level Security on the `vault` table before applying any RLS policy; the service-role-only policy SHALL only be considered successfully applied after RLS is confirmed enabled on that table.
12. WHEN the Migration_File has been applied to a Supabase project, THE Supabase_Client SHALL be able to insert and query rows in all tables without schema errors; IF the Migration_File has not been applied, THEN all database insert and query operations SHALL fail with a schema-not-found error rather than silently succeeding.

### Requirement 2: Supabase Persistence Layer — Replacing Filesystem Stores

**User Story:** As an Admin, I want all runtime write operations to target Supabase instead of the local filesystem, so that data is durable across deployments and accessible from any Vercel function instance.

#### Acceptance Criteria

1. WHEN the Clearance_Gate appends an audit log entry, THE Codex SHALL insert a row into the `audit_log` Supabase table instead of writing to `/vault/AUDIT_LOG.jsonl`.
2. WHEN the Loop Engine records a loop execution, THE Codex SHALL insert a row into the `loop_execution_log` Supabase table instead of appending to `/core/LOOP_EXECUTION_LOG.jsonl`.
3. WHEN the Strike Output Engine persists a generated Instrument, THE Codex SHALL insert a row into both the `instruments` table and the `instrument_registry` table instead of writing to the `/instruments/` directory.
4. WHEN a capital allocation is approved, THE Codex SHALL insert a row into the `capital_allocations` Supabase table instead of appending to `/core/CAPITAL_ALLOCATIONS.json`.
5. WHEN a capital allocation fails validation, THE Codex SHALL insert a row into the `capital_allocation_failures` Supabase table instead of appending to `/core/CAPITAL_ALLOCATION_FAILURES.json`.
6. WHEN the Vault is read or written, THE Codex SHALL use the `vault` Supabase table via the Supabase_Service_Client instead of reading or writing `/vault/CLEARANCE_CODES.json`.
7. THE Codex SHALL initialise the Supabase_Client as a singleton using `SUPABASE_URL` and `SUPABASE_ANON_KEY` environment variables, and the Supabase_Service_Client using `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`.
8. IF a Supabase write operation fails, THEN THE Codex SHALL retry the operation up to two additional times with exponential back-off before returning a structured error using the existing `{ error: { code, message, details? } }` envelope; THE Codex SHALL NOT silently discard the failure after all retries are exhausted.
9. WHILE the in-memory store is populated at startup, THE Codex SHALL continue to load Pillars, OS Modules, and Library entries from Markdown files; these entities SHALL NOT be migrated to Supabase.
10. THE Codex SHALL expose a `SupabaseRepository` module at `src/lib/db/supabase.ts` that exports the singleton Supabase_Client and Supabase_Service_Client instances.

### Requirement 3: AI Chat Assistant

**User Story:** As a User, I want a streaming AI chat assistant embedded in the dashboard, so that I can query Codex data and receive strategic guidance through a conversational interface without leaving the application.

#### Acceptance Criteria

1. THE Chat_Widget SHALL be accessible from the dashboard to all authenticated users regardless of clearance level.
2. WHEN a User submits a message in the Chat_Widget, THE Chat_Widget SHALL stream the assistant's response token-by-token using the Vercel AI SDK `streamText` API and display tokens as they arrive.
3. THE Chat_Widget SHALL use the OpenAI `gpt-4o` model via the Vercel AI SDK.
4. THE Chat_Widget SHALL expose the following AI_Tools to the model during each turn:
   a. `getPillar(code: string)` — returns the full Pillar record for the given code.
   b. `searchPillars(query: string)` — returns the top-10 Pillar search results for the given query.
   c. `getOSModule(slug: string)` — returns the full OS Module record for the given slug.
   d. `searchLibrary(query: string)` — returns the top-10 Library search results for the given query.
   e. `getIntegrationMap(slug: string)` — returns the inbound and outbound Integration records for the given OS Module slug.
   f. `getLoopStatus(loopId: string)` — returns the most recent execution record for the given loop ID from the `loop_execution_log` table.
   g. `getInstrumentSummary(year: number)` — returns the list of Instrument registry entries for the given year from the `instrument_registry` table.
   h. `getCapitalAllocationSummary()` — returns the most recent approved capital allocation record from the `capital_allocations` table.
5. WHEN the model invokes an AI_Tool, THE Chat_Widget SHALL execute the corresponding Codex service call on the server and return the result to the model before continuing generation.
6. WHEN a User starts a new conversation, THE Chat_Widget SHALL create a new Chat_Session row in the `agent_conversations` table and associate all subsequent messages with that session.
7. WHEN a message exchange completes, THE Chat_Widget SHALL persist both the user message and the assistant response as Chat_Message rows in the `agent_messages` table, including any tool call payloads in the `tool_calls` column.
8. WHEN a User returns to the dashboard, THE Chat_Widget SHALL load and display the most recent Chat_Session for that user, including all prior Chat_Messages in the correct order; IF loading the most recent session fails due to a database error or corrupted session data, THEN THE Chat_Widget SHALL silently fall back to creating a new empty Chat_Session so the widget remains functional.
9. IF the OpenAI API returns an error during streaming, THEN THE Chat_Widget SHALL display a descriptive error message to the User and SHALL NOT leave the UI in a loading state.
10. THE Chat_Widget SHALL be implemented as a client-side React component at `src/components/ui/ChatWidget.tsx` with a corresponding API route at `src/app/api/agent/chat/route.ts`.
11. THE Chat_Widget API route SHALL be protected by the existing clearance gate middleware and SHALL return a 401 response for unauthenticated requests.

### Requirement 4: Autonomous Background Agent

**User Story:** As an Admin, I want an autonomous background agent that evaluates all Recursive Loop conditions on a schedule and generates daily summaries, so that the system self-monitors and self-triggers without requiring manual intervention.

#### Acceptance Criteria

1. THE Background_Agent SHALL be invoked by a Vercel_Cron job defined in `vercel.json` at a schedule of every 15 minutes (`*/15 * * * *`).
2. WHEN the Background_Agent runs, THE Background_Agent SHALL evaluate all 104 Recursive_Loop definitions from the in-memory store against the current system state.
3. WHEN a Loop_Evaluation determines that a Recursive_Loop's trigger condition is satisfied, THE Background_Agent SHALL check whether that loop has already been triggered by another system component within the current 15-minute window by querying the `loop_execution_log` table; IF the loop was already triggered, THE Background_Agent SHALL skip it and set `triggered_by` to `'background-agent-skipped'` in the log entry; IF the loop has not been triggered, THE Background_Agent SHALL invoke the Loop Engine's trigger method and record the execution with `triggered_by` set to `'background-agent'`.
4. THE Background_Agent SHALL insert a run-level summary row into the `loop_execution_log` table at the end of every cron execution, regardless of whether any loops triggered, recording the total number of loops evaluated, the number triggered, and the number skipped.
5. THE Background_Agent SHALL be invoked by a second Vercel_Cron job at `0 6 * * *` (06:00 UTC daily) to generate a Daily_Summary Strike Output.
6. WHEN the daily cron fires, THE Background_Agent SHALL generate a Strike Output using the AI-Powered Strike Output Engine (Requirement 6), passing a Structured_Context that includes the current state of all 36 OS Modules, the most recent capital allocation, and any loops triggered in the preceding 24 hours.
7. WHEN the Daily_Summary is generated, THE Background_Agent SHALL persist it to the `instruments` and `instrument_registry` Supabase tables with `generated_by` set to `'background-agent'`.
8. WHEN a Critical_Loop is triggered by the Background_Agent, THE Background_Agent SHALL send a Webhook_Alert to the URL configured in the `WEBHOOK_ALERT_URL` environment variable within 30 seconds of the trigger.
9. THE Webhook_Alert payload SHALL be a JSON object containing: `loopId` (string), `loopTitle` (string), `triggeredAt` (ISO 8601 UTC string), `targetSlug` (string), `outputAction` (string), and `severity` (string `'critical'`).
10. IF the `WEBHOOK_ALERT_URL` environment variable is not set, THEN THE Background_Agent SHALL log a warning and skip the webhook dispatch without failing the cron execution.
11. IF the webhook endpoint returns a non-2xx HTTP status, THEN THE Background_Agent SHALL log the failure with the status code and response body and continue processing remaining loops.
12. THE Background_Agent SHALL be implemented as a Next.js API route at `src/app/api/agent/cron/route.ts` and SHALL verify the `Authorization: Bearer {CRON_SECRET}` header before executing, returning 401 if the header is absent or incorrect.
13. THE Background_Agent cron route SHALL be protected from public access by requiring the `CRON_SECRET` environment variable to match the `Authorization` header sent by Vercel's cron scheduler.

### Requirement 5: Conversation History Management

**User Story:** As a User, I want my chat conversations with the AI assistant to be persisted and retrievable, so that I can continue prior conversations and review past strategic guidance.

#### Acceptance Criteria

1. THE Codex SHALL expose a `GET /api/agent/conversations` route that returns all Chat_Sessions for the authenticated user, ordered by `updated_at` descending.
2. WHEN a User requests a specific conversation, THE Codex SHALL expose a `GET /api/agent/conversations/[id]` route that returns the Chat_Session record and all associated Chat_Messages ordered by `created_at` ascending.
3. WHEN a User sends a message in an existing Chat_Session, THE Chat_Widget SHALL append new Chat_Messages to the existing session rather than creating a new Chat_Session.
4. THE Codex SHALL expose a `DELETE /api/agent/conversations/[id]` route that deletes the specified Chat_Session and all associated Chat_Messages (via cascade) for the authenticated user.
5. IF a User requests a Chat_Session that does not belong to their user ID, THEN THE Codex SHALL return a 403 structured error with code `CONVERSATION_ACCESS_DENIED`.
6. THE `GET /api/agent/conversations` and `GET /api/agent/conversations/[id]` routes SHALL be protected by the existing clearance gate middleware and SHALL return 401 for unauthenticated requests.

### Requirement 6: AI-Powered Strike Output Engine (Claude)

**User Story:** As a User, I want the Strike Output Engine to use Claude to generate all five sections with real strategic reasoning, so that outputs are substantive and contextually grounded rather than template-filled.

#### Acceptance Criteria

1. WHEN a Strike Output is requested, THE Claude_Engine SHALL receive a Structured_Context payload containing: the requesting user's query or intent, the top-5 relevant Pillars (by search relevance), the top-3 relevant OS Modules (by search relevance), the top-3 relevant Library entries (by search relevance), and the most recent approved capital allocation record.
2. WHEN the Claude_Engine generates a Strike Output, THE Claude_Engine SHALL use the Anthropic `claude-3-5-sonnet-20241022` model via `@anthropic-ai/sdk`.
3. WHEN the Claude_Engine returns a response, THE Strike_Validation process SHALL verify that the response contains exactly five sections with the labels: `## Executive Analysis`, `## OS Stress Test`, `## The Imperial Instrument`, `## Action Plan (T-Minus 24 Hours)`, and `## The Ritual`, in that order.
4. IF Strike_Validation fails (missing sections, wrong labels, wrong order), THEN THE Codex SHALL discard the Claude response and invoke the Template_Fallback to generate the Strike Output.
5. IF the Anthropic API is unavailable (network error, timeout, or HTTP 5xx response), THEN THE Codex SHALL invoke the Template_Fallback within 5 seconds of the failure and SHALL log the fallback event with the error details.
6. WHEN the Template_Fallback is invoked, THE Codex SHALL generate the Strike Output using the existing template-based Strike Output Engine logic from Imperial Codex v16.
7. WHEN a Claude-generated Strike Output passes Strike_Validation, THE Codex SHALL persist it to the `instruments` and `instrument_registry` Supabase tables with `generated_by` set to `'claude-engine'`.
8. WHEN a Template_Fallback-generated Strike Output is persisted, THE Codex SHALL set `generated_by` to `'template-fallback'` in the `instruments` table.
9. THE Claude_Engine SHALL be implemented in `src/lib/strike/ClaudeStrikeEngine.ts` and SHALL be invoked by the existing `StrikeOutputEngine` orchestrator before falling back to the template path.
10. THE Structured_Context assembled for Claude SHALL not exceed 100,000 tokens; IF the assembled context would exceed this limit, THEN THE Codex SHALL truncate Library entry bodies to 500 characters each before sending.

### Requirement 7: Strike Output Structural Validation

**User Story:** As an Admin, I want every AI-generated Strike Output to be validated against the 5-Part Strike Hierarchy structure before being accepted, so that no malformed output is persisted or presented to users.

#### Acceptance Criteria

1. THE Strike_Validation process SHALL check that the generated output contains exactly five section headings at the `##` level.
2. THE Strike_Validation process SHALL check that the five section headings appear in the exact order: `Executive Analysis`, `OS Stress Test`, `The Imperial Instrument`, `Action Plan (T-Minus 24 Hours)`, `The Ritual`.
3. THE Strike_Validation process SHALL check that each section body contains at least 50 characters of non-whitespace content.
4. IF any of the three validation checks fail, THEN THE Strike_Validation process SHALL return a structured validation result identifying which checks failed and SHALL trigger the Template_Fallback.
5. THE Strike_Validation process SHALL be implemented as a pure function in `src/lib/strike/StrikeValidator.ts` that accepts a string and returns `{ valid: boolean, failures: string[] }`.
6. WHEN Strike_Validation passes, THE Codex SHALL proceed to persist the output and return it to the caller.

### Requirement 8: Vercel Deployment Configuration

**User Story:** As an Admin, I want a complete Vercel deployment configuration with cron jobs, environment variable documentation, and Supabase migration files, so that the application can be deployed to Vercel and connected to Supabase without manual configuration steps.

#### Acceptance Criteria

1. THE Codex SHALL provide a `vercel.json` file at the repository root that defines the two Cron_Jobs: one at schedule `*/15 * * * *` targeting `/api/agent/cron` and one at schedule `0 6 * * *` targeting `/api/agent/cron`.
2. THE `vercel.json` file SHALL set the cron job region to `iad1` (US East) to minimise latency to Supabase.
3. THE Codex SHALL update `.env.example` to document all required environment variables: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `VAULT_ENCRYPTION_KEY`, `SESSION_SECRET`, `CRON_SECRET`, and `WEBHOOK_ALERT_URL` (optional).
4. THE `.env.example` file SHALL include a descriptive comment for each variable explaining its purpose, expected format, and where to obtain the value.
5. THE Codex SHALL remove `netlify.toml` from the repository root, as Netlify is no longer the deployment target.
6. THE `/supabase/migrations/` directory SHALL contain at least one Migration_File named with the format `YYYYMMDDHHMMSS_description.sql` (e.g., `20250101000000_initial_schema.sql`).
7. WHEN the `vercel.json` cron configuration is deployed, Vercel SHALL invoke the `/api/agent/cron` route at the configured schedules with an `Authorization: Bearer {CRON_SECRET}` header.
8. THE Codex SHALL provide a `README` section or a `/docs/deployment.md` file documenting the steps to: create a Supabase project, run migrations, set environment variables in Vercel, and deploy.

### Requirement 9: Backward Compatibility with In-Memory Store

**User Story:** As a Developer, I want all existing read operations against Pillars, OS Modules, and Library entries to continue working without modification, so that the Supabase migration does not break any existing functionality.

#### Acceptance Criteria

1. THE Codex SHALL continue to load all 207 Pillars from `/core/PILLARS.md` into the in-memory store at startup; Pillars SHALL NOT be read from Supabase.
2. THE Codex SHALL continue to load all 36 OS Modules from `/os-modules/*.md` into the in-memory store at startup; OS Modules SHALL NOT be read from Supabase.
3. THE Codex SHALL continue to load all 345 Library entries from `/core/LIBRARY.md` into the in-memory store at startup; Library entries SHALL NOT be read from Supabase.
4. WHEN an AI_Tool in the Chat_Widget queries Pillars, OS Modules, or Library entries, THE AI_Tool SHALL read only the data type it requires from the in-memory store; each AI_Tool is permitted to query a single data type and is not required to query all three types simultaneously.
5. THE existing API routes (`/api/pillars`, `/api/os-modules`, `/api/library`, `/api/integrations`, `/api/loops`) SHALL continue to function without modification after the Supabase migration.
6. THE existing clearance gate middleware, session management, and Kernel loader SHALL continue to function without modification after the Supabase migration.

### Requirement 10: Error Handling and API Consistency

**User Story:** As a Developer, I want all new API routes to follow the existing error envelope convention and be protected by the clearance gate, so that the AI agent additions are consistent with the rest of the Codex API surface.

#### Acceptance Criteria

1. THE Codex SHALL return all errors from new API routes in the existing `{ error: { code, message, details? } }` JSON envelope.
2. THE Codex SHALL protect all new API routes (`/api/agent/chat`, `/api/agent/conversations`, `/api/agent/conversations/[id]`) with the existing clearance gate middleware.
3. IF a Supabase query returns an error, THEN THE Codex SHALL map it to a structured error response with a machine-readable code (e.g., `DB_QUERY_FAILED`, `DB_INSERT_FAILED`) and SHALL NOT expose raw Supabase error messages to the client.
4. THE Codex SHALL add the following error codes to the error code registry:
   - `DB_QUERY_FAILED` (500) — Supabase SELECT operation failed.
   - `DB_INSERT_FAILED` (500) — Supabase INSERT operation failed.
   - `AGENT_STREAM_ERROR` (500) — OpenAI streaming API returned an error.
   - `CLAUDE_API_UNAVAILABLE` (503) — Anthropic API is unreachable; fallback was invoked.
   - `CLAUDE_VALIDATION_FAILED` (422) — Claude response failed Strike_Validation; fallback was invoked.
   - `CONVERSATION_NOT_FOUND` (404) — Requested Chat_Session does not exist.
   - `CONVERSATION_ACCESS_DENIED` (403) — Requested Chat_Session belongs to a different user.
   - `CRON_UNAUTHORIZED` (401) — Background_Agent cron route called without valid `CRON_SECRET`.
   - `WEBHOOK_DISPATCH_FAILED` (n/a, log only) — Webhook endpoint returned non-2xx; execution continues.
5. WHEN the Background_Agent cron route is called without a valid `Authorization` header, THE Codex SHALL return a 401 response with code `CRON_UNAUTHORIZED` and SHALL NOT execute any loop evaluations.
